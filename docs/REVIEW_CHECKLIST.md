# 工程化设计审核清单

> 适用于"设计阶段完成、正式编码前"的 Gate Review。
> 每个检查项标注 [必须] 或 [推荐]，未通过 [必须] 项不应进入编码阶段。

---

## 一、项目立项与决策（EVALUATION 层）

### 1.1 动机与价值 [必须]

- [ ] 问题定义清晰（解决谁的什么问题）
- [ ] 不做的后果已说明（机会成本）
- [ ] 成功标准可量化（Definition of Done）
- [ ] 与现有方案的差异化已分析（竞品对比）

### 1.2 可行性验证 [必须]

- [ ] 核心技术假设已列出并标注验证状态（✅/⏳/❌）
- [ ] 关键假设有实测数据支撑（非纯理论推断）
- [ ] 依赖项（外部服务/库/工具）的稳定性已评估
- [ ] 依赖项的更新频率和 breaking change 风险已分析

### 1.3 决策完备性 [必须]

- [ ] 所有已识别的设计选择都有明确决策（YES/NO + 归属期）
- [ ] 每个决策有理由记录
- [ ] 待决策项已列出，含触发条件和预期决策时间
- [ ] 不存在"隐含假设"——所有假设已显式化

### 1.4 风险评估 [必须]

- [ ] 已列出项目风险（概率 × 影响矩阵）
- [ ] 每个高风险项有缓解措施
- [ ] 有降级/回退方案（graceful degradation）

---

## 二、架构与技术设计（DESIGN 层）

### 2.1 架构清晰度 [必须]

- [ ] 总体架构图存在且可读
- [ ] 模块划分有明确的单一职责
- [ ] 数据流（启动 + 运行时）已图示化
- [ ] 外部依赖边界清晰（哪些是信任边界）

### 2.2 模块接口 [必须]

- [ ] 模块间调用契约已定义（输入类型、输出类型）
- [ ] 错误传播策略明确（异常层级 / 错误码 / 返回值）
- [ ] 状态管理策略明确（有状态 vs 无状态、生命周期）

### 2.3 协议与通信 [必须]

- [ ] 对外协议细节已定义（transport、消息格式、分隔符）
- [ ] 进程/线程模型已明确（单进程?多进程?协程?）
- [ ] 超时与重试策略已设计

### 2.4 降级与容错 [必须]

- [ ] 各种故障场景下的行为已定义（依赖不可用、认证失效、超时等）
- [ ] 降级状态有用户可感知的反馈（非静默失败）
- [ ] 最低依赖版本要求已明确

### 2.5 性能设计 [推荐]

- [ ] 已有性能 baseline 数据或预期目标
- [ ] 瓶颈点已识别并有优化路径（即使当前不实施）
- [ ] 缓存策略已设计（如有）

### 2.6 可扩展性 [推荐]

- [ ] 已标注未来扩展点（不过度设计，但留口子）
- [ ] 扩展时的改动范围可预估

---

## 三、接口契约（API_SPEC 层）

### 3.1 外部接口 [必须]

- [ ] 所有对外接口有完整 schema 定义
- [ ] 请求/响应格式有示例
- [ ] 错误码枚举完整，每个错误有 recovery_hint
- [ ] 接口版本化策略明确

### 3.2 语言与命名 [必须]

- [ ] 命名规范已确定且一致（camelCase / kebab-case 等）
- [ ] 人类可读内容的语言策略已决策
- [ ] 示例能覆盖主要使用场景

---

## 四、安全（SECURITY 层）

### 4.1 权限模型 [必须]

- [ ] 默认策略是 deny 还是 allow 已明确
- [ ] 权限控制粒度已定义
- [ ] 高风险操作有二次确认机制

### 4.2 数据安全 [必须]

- [ ] 敏感数据（凭证、token）的存储和传输方式已确认
- [ ] 日志中的敏感数据有脱敏规则
- [ ] bridge 自身不持有/缓存用户凭证（如适用）

### 4.3 威胁模型 [推荐]

- [ ] 已列出主要威胁向量
- [ ] 每个威胁有对应缓解措施

---

## 五、可观测性（OBSERVABILITY 层）

### 5.1 运维可见性 [推荐]

- [ ] 健康检查端点已设计
- [ ] 关键指标已定义（延迟、错误率、吞吐）
- [ ] 日志级别和格式已规范
- [ ] 启动时有诊断输出（版本、配置摘要、依赖状态）

### 5.2 问题排查 [推荐]

- [ ] 错误有唯一追踪 ID（trace_id）
- [ ] 错误分类体系可支撑快速定位

---

## 六、测试策略（TESTING 层）

### 6.1 可测试性 [必须]

- [ ] 外部依赖可 Mock（有注入点）
- [ ] 测试分层策略已定义（单元/集成/E2E）
- [ ] CI 可在无认证环境下运行测试

### 6.2 测试覆盖 [推荐]

- [ ] 各模块的测试重点已明确
- [ ] Fixture/测试数据的来源和维护方式已定义
- [ ] 覆盖率目标已设定

---

## 七、工程规范

### 7.1 项目结构 [必须]

- [ ] 目录结构已定义
- [ ] 依赖管理方式确定（pyproject.toml / requirements.txt 等）
- [ ] License 已选择

### 7.2 版本管理 [必须]

- [ ] 版本号策略确定（SemVer + breaking change 定义）
- [ ] CHANGELOG 维护规范确定
- [ ] Commit message 规范确定（Conventional Commits 或其他）
- [ ] 分支策略确定（GitHub Flow / 直推 main / 其他）
- [ ] 文档与代码的仓库关系确定（同仓 / 分离）

### 7.3 版本控制实践 [必须]

- [ ] `.gitignore` 覆盖完整（缓存、编译产物、.env、IDE 配置、fixture 中的敏感数据）
- [ ] 设计阶段有"设计基线" tag（如 `v0.0.0-design`），便于后续 diff
- [ ] 代码变更涉及接口时，同一 commit 同步更新相关文档

### 7.4 版本控制审核（跨生命周期） [推荐]

- [ ] commit 历史可追溯——每个设计变更有独立 commit（非"一次性提交全部"）
- [ ] commit message 能反映意图（type + scope + 描述）
- [ ] tag 策略明确（何时打 tag、命名规则）
- [ ] 版本发布时 CHANGELOG 与 tag 同步
- [ ] 有 PR 模板或 commit checklist 提醒文档同步（开源后启用）

### 7.5 文档体系 [推荐]

- [ ] 文档索引存在，职责边界清晰
- [ ] 各文档的目标受众已标注
- [ ] 交叉引用约定一致

---

## 八、迭代计划（ROADMAP 层）

### 8.1 分期合理性 [必须]

- [ ] MVP 范围足够小（可在 1-2 周内完成）
- [ ] 各 Phase 有独立可验证的交付物
- [ ] 关键验证点在早期（fail fast 原则）

### 8.2 里程碑 [推荐]

- [ ] 有验证性里程碑（Phase X.5）穿插在功能迭代之间
- [ ] 工时估算已给出（即使粗略）
- [ ] 依赖关系明确（哪个 Phase 依赖哪个）

---

## 九、跨文档质量（文档体系整体性）

### 9.1 一致性 [必须]

- [ ] 同一概念在所有文档中使用相同术语（如 tool name 规则只定义一次，其他引用）
- [ ] 模块名/文件名在架构图、目录结构、接口定义中一致
- [ ] 环境变量名在 DESIGN、DEPLOYMENT、OBSERVABILITY 中一致
- [ ] 错误码在 API_SPEC、SECURITY、OBSERVABILITY 中定义一致
- [ ] Phase 编号和内容在 ROADMAP 与 EVALUATION 决策归属期中一致

### 9.2 无冲突 [必须]

- [ ] 不存在相互矛盾的描述（如一处写"白名单优先"另一处写"默认允许"）
- [ ] 决策项之间无逻辑冲突（如决策 A 的前提与决策 B 的结论矛盾）
- [ ] 数字/数据引用一致（如 tool 数量、超时值、端口号各处相同）

### 9.3 无冗余 [必须]

- [ ] 同一设计细节只在一个文档中**定义**，其他文档通过引用链接
- [ ] 不存在大段复制粘贴的内容（应提取为引用）
- [ ] 每个文档在 INDEX 职责边界内，不越界包含其他文档应承载的内容

### 9.4 引用完整 [必须]

- [ ] DESIGN 中的每个设计决策都能在 EVALUATION 中找到对应决策条目
- [ ] ROADMAP 中的每个 TODO 都能追溯到 EVALUATION 的已决策项
- [ ] 跨文档引用链接有效（相对路径、锚点存在）
- [ ] 不存在"悬空设计"——DESIGN 中有但 EVALUATION 无决策依据的内容

### 9.5 覆盖完整 [必须]

- [ ] EVALUATION 中的所有"已决策 YES"项在 DESIGN 或其他实现文档中有体现
- [ ] 不存在"遗忘的决策"——决策了但没有落实到设计/实现文档中
- [ ] 所有文档在 INDEX 中有索引
- [ ] 新增的模块/环境变量/错误码在所有相关文档中同步更新

### 9.6 时效性 [推荐]

- [ ] 不存在过期信息（如"计划中"的功能实际已完成但文档未更新）
- [ ] 验证数据标注了日期和版本（如"lark-cli v1.0.43, 2026-05-29"）
- [ ] 待决策项未过期（触发条件是否已满足但未处理）

### 9.7 可维护性 [推荐]

- [ ] 修改一个设计决策时，涉及的文档列表可快速确定（通过 INDEX 职责边界）
- [ ] 文档间耦合度低（改 SECURITY 不需要同步改 DEPLOYMENT 的正文）
- [ ] 引用方向单一（DESIGN 引用 EVALUATION，EVALUATION 不反向引用 DESIGN 的实现细节）

---

## 使用方式

1. **设计完成后**：逐项过一遍，标记状态
2. **[必须] 项全部通过** → 可以开始编码
3. **[推荐] 项部分未通过** → 记录为技术债务，不阻塞编码
4. **发现新维度** → 更新本清单（清单本身是活文档）
